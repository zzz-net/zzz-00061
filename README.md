# 档案移交流转台

本地档案移交流转系统，支持批次登记、装盒、移交、签收、退回、导出全流程可追溯。

## 快速启动

### 1. 安装依赖

```bash
pip install -r requirements.txt
```

### 2. 启动服务

```bash
python app.py
```

### 3. 访问系统

打开浏览器访问：http://127.0.0.1:5000/

## 默认账号

系统初始化时自动创建两个测试账号：

- `sender` - 发送方（负责登记、装盒、发起移交）
- `receiver` - 接收方（负责签收、退回、撤销退回）

在页面顶部的"当前用户"下拉菜单中切换身份。

## 操作流程

### 主流程（正常移交）

#### 步骤1：登记批次（发送方操作）

1. 点击"登记新批次"标签页
2. 输入**批次号**（全局唯一，如：DA-2025-001）
3. 输入批次描述（可选）
4. 添加档案：逐条输入**档号**（同一批次内不能重复）、**题名**、备注
5. 点击"登记批次"按钮

> ⚠️ 校验规则：
> - 批次号重复会被拒绝
> - 同一批次内档号重复会被拒绝，登记失败

#### 步骤2：装盒（发送方操作）

1. 在批次列表中点击刚创建的批次，进入详情页
2. 点击"+ 创建档案盒"，输入**盒号**（全局唯一，如：BOX-001）
3. 在档案盒卡片中勾选要装入的档案
4. 点击"确认装盒"
5. 可创建多个档案盒，分别装入不同档案
6. 当所有档案都装入盒子后，批次状态自动变为"已装盒"

> ⚠️ 校验规则：
> - 已装入其他盒子的档案不能再勾选
> - 移交后不能再修改装盒内容

#### 步骤3：发起移交（发送方操作）

1. 确认所有档案都已装盒
2. 点击"→ 发起移交"按钮
3. 确认提示后，批次和所有盒子状态变为"已移交"

#### 步骤4：签收（接收方操作）

1. **切换当前用户为 `receiver`（接收方）**
2. 进入批次详情页
3. 对每个档案盒点击"✓ 签收"按钮
4. 所有盒子签收完成后，批次状态变为"已签收"

> ⚠️ 校验规则：
> - 发送方不能代替接收方签收，会被拒绝
> - 必须使用接收方账号才能签收

#### 步骤5：导出移交清单

1. 点击"📥 导出移交清单"按钮
2. 自动下载 CSV 格式的移交清单文件
3. 导出记录会保存在"导出记录"区域，点击"重新下载"可随时下载对应时刻的历史快照，每份记录按导出记录独立编号存储，不会和其他批次或其他导出记录串号
4. 主批次"📥 导出移交清单"按钮下载的是该批次当前最新状态

导出清单包含：
- 批次基本信息（批次号、描述、创建人、状态、创建时间）
- 档案详情（档号、题名、备注、盒号、盒子状态）
- 完整流转历史（操作、操作人、时间、盒号、原因）

### 失败链路1：退回与撤销退回

#### 退回（接收方操作）

1. **可退回的状态**：盒子处于"已移交"或"已签收"状态时，接收方都可以选择退回
2. 点击"✗ 退回"按钮（已签收的盒子显示为"✗ 退回（签收后也可退）"）
3. **必须填写退回原因**（不能为空）
4. 确认后，盒子状态变为"已退回"，批次状态也变为"已退回"
5. **退回前快照自动保存**：系统会自动记录退回前的真实状态（已移交/已签收）、签收人、签收时间，写入流转历史，撤销退回时可精确还原

#### 撤销退回（接收方操作）

1. 盒子处于"已退回"状态时，可以撤销退回
2. 点击"↩ 撤销退回，恢复原状态"
3. **根据退回前快照精确恢复**：
   - 如果退回前是"已签收"：盒子恢复为"已签收"状态，签收人和原始签收时间都会还原，批次状态也会相应恢复
   - 如果退回前是"已移交"：盒子恢复为"已移交"状态，可重新签收
4. 批次状态智能聚合：
   - 如果所有盒子都撤销后都是"已签收" → 批次变为"已签收"
   - 如果有盒子是"已移交"或混合状态 → 批次变为"已移交"
   - 如果还有其他盒子是"已退回" → 批次保持"已退回"

> ⚠️ 校验规则：
> - 只有接收方才能退回和撤销退回
> - 退回原因不能为空
> - 撤销退回不会丢失签收记录，签收人、签收时间都从快照恢复

### 失败链路2：重复档号校验

在以下场景会校验重复档号：
1. 登记批次时，同一批次内有相同档号 → 登记被拒绝
2. 向已登记批次添加档案时，档号已存在 → 添加被拒绝

## 数据持久化

系统使用 SQLite 数据库（`archive_transfer.db` 文件）存储所有数据：

- ✅ 批次信息
- ✅ 档案信息
- ✅ 档案盒信息
- ✅ 档案与盒号映射关系
- ✅ 完整流转历史（经手人、时间戳、盒号、原因）
- ✅ 签收记录
- ✅ 退回原因
- ✅ 导出清单内容

**断电或重启后，所有数据完整保留**，重新启动服务即可继续操作，数据不会丢失。

---

## 流转历史记录

每个操作都会记录完整历史，包括：
- 操作类型（登记、装盒、移交、签收、退回、撤销退回、导出、新建复核项、更新复核项、退回复核项、关闭复核项、撤销重开复核项、发起催办、处理催办、撤销升级）
- 经手人及角色
- 精确时间戳
- 涉及的盒号
- 操作原因/备注

在批次详情页底部的时间线中可查看完整历史。

## 状态流转说明

### 批次状态

| 状态 | 说明 |
|------|------|
| 已登记 | 批次已创建，可添加档案和装盒 |
| 已装盒 | 所有档案已装入盒子，可发起移交 |
| 已移交 | 已发起移交，等待接收方处理 |
| 已签收 | 所有盒子已签收，流程完成 |
| 已退回 | 有盒子被退回（退回前可能是已移交或已签收） |

### 档案盒状态

| 状态 | 说明 | 可执行操作 |
|------|------|----------|
| 空盒 | 已创建，未装入档案 | 勾选档案、装盒 |
| 已装盒 | 已装入档案 | 等待移交 |
| 已移交 | 已发起移交 | 签收或退回（接收方操作） |
| 已签收 | 已被接收方签收 | 仍可退回（签收后也能退回，接收方操作） |
| 已退回 | 已被接收方退回 | 撤销退回，恢复退回前的真实状态 |

> **状态流转补充说明**：
> - 已移交 → 已签收（签收）
> - 已移交 → 已退回（退回）
> - 已签收 → 已退回（退回，签收后也可退）
> - 已退回 → 已移交（撤销退回，退回前是已移交的情况）
> - 已退回 → 已签收（撤销退回，退回前是已签收的情况，签收人、签收时间精确还原）

## 失败链路3：签收后异常复核（复核单管理）

批次到了"已签收"状态后，如果遇到**盒内材料缺页、标签不清、签收后补件**等异常情况，不再只能靠退回备注硬扛，可以通过「复核单」按盒子登记问题，全流程跟踪处理进度。

### 复核单字段

| 字段 | 说明 |
|------|------|
| 档案盒 | 问题发生在哪个盒子（必须从该批次的已有盒子中选择） |
| 问题类型 | 预设选项：材料缺页 / 标签不清 / 签收后补件 / 顺序混乱 / 其他问题 |
| 问题描述 | 详细描述问题，包括涉及的档号、具体情况 |
| 责任方 | 可选，指定负责处理该问题的部门/人员 |
| 截止时间 | 可选，要求处理完成的截止日期 |
| 处理说明 | 发送方填写的处理进展和处理结果 |
| 状态 | 待处理 → 处理中 → 申请关闭 → 已关闭；或 任何状态 → 已退回（需重处理） |

### 复核单状态流转与权限

**角色分工严格不交叉：**

| 操作 | 权限角色 | 说明 |
|------|---------|------|
| 新建复核项 | 接收方 | 按盒子登记问题，初始状态为「待处理」 |
| 更新处理结果 | 发送方 | 可改为「处理中」或「申请关闭」；申请关闭前必须填写处理说明 |
| 退回复核项 | 接收方 | 不认可处理结果时退回，状态变为「已退回」，退回原因必填 |
| 确认关闭 | 接收方 | 只有「申请关闭」状态才能确认关闭，状态变为「已关闭」 |
| 撤销重开 | 接收方 | 「已关闭」的复核项可撤销重开，原因必填；状态恢复为「待处理」，closed_by/closed_at 清空 |

**状态流转图：**

```
待处理(OPEN) ──发送方更新──► 处理中(IN_PROGRESS) ──发送方申请关闭──► 申请关闭(PENDING_CLOSE)
     │                              │                              │
     │                              │                              │接收方确认关闭
     │                              │                              ▼
     │退回                          │退回                       已关闭(CLOSED)
     │                              │                              │
     ▼                              ▼                              │接收方撤销重开
  已退回(REJECTED) ◄───────────────┘                               │
     │                                                              │
     └────────────── 发送方可重新更新为 处理中/申请关闭 ◄───────────┘
```

### 操作步骤

#### 步骤1：接收方新建复核项（接收方操作）

1. **切换当前用户为 `receiver`（接收方）**
2. 进入批次详情页，找到底部「📋 复核单管理」卡片
3. 点击右上角「+ 新建复核项」
4. 选择**档案盒**（该批次的所有已签收/已移交盒子都可选）
5. 选择**问题类型**，填写**问题描述**（必填）
6. 可选填写**责任方**和**处理截止时间**
7. 点击「提交复核项」

> ⚠️ 校验规则：
> - 只有接收方才能新建复核项，发送方会被拒绝
> - 问题类型 + 问题描述在同一盒子下重复且未关闭时，会返回 **409 冲突**，提示已有相同问题
> - 同一盒子下的不同问题、不同盒子的相同问题都允许正常提交
> - 提交后自动写入时间线（流转历史）

#### 步骤2：发送方更新处理结果（发送方操作）

1. **切换当前用户为 `sender`（发送方）**
2. 在复核单卡片中找到对应的复核项
3. 点击「✏️ 更新处理结果」
4. 选择更新后的状态：
   - **处理中**：正在处理，不需要接收方回应
   - **申请关闭**：已处理完成，等待接收方确认
5. 填写**处理说明**（申请关闭为必填）
6. 点击「提交更新」

> ⚠️ 校验规则：
> - 只有发送方才能更新处理结果，接收方会被拒绝
> - 已关闭的复核项不能直接更新，需要先撤销重开
> - 申请关闭必须填写处理说明，否则会被拒绝

#### 步骤3：接收方确认关闭 / 退回（接收方操作）

**确认关闭：**
1. 复核项状态为「申请关闭待确认」时，点击「✓ 确认关闭」
2. 确认后状态变为「已关闭」，记录关闭人和关闭时间

**退回（不认可处理）：**
1. 点击「✗ 退回（不认可处理）」
2. **必填退回原因**：说明不认可的地方或需要补充的内容
3. 状态变为「已退回（需重新提报）」，发送方需要重新更新处理结果

#### 步骤4：撤销重开已关闭项（接收方操作）

1. 找到已关闭的复核项，点击「↩ 撤销重开」
2. **必填撤销原因**：说明为什么需要重新处理
3. 状态恢复为「待处理」，关闭人和关闭时间清空
4. 发送方可以重新开始处理流程

> 💡 撤销重开后，完整的历史记录仍然保留在时间线中，可以追溯完整过程。

### 盒子卡片上的复核摘要

每个档案盒卡片的标题下方会显示复核摘要徽章：
```
📋 复核: 2待处理/1已关闭
```
- 待处理数 = 所有 OPEN + IN_PROGRESS + PENDING_CLOSE + REJECTED 状态的复核项
- 已关闭数 = CLOSED 状态的复核项
- 点击可跳转到复核单管理区查看详情

### 导出清单包含复核信息

导出移交清单时，自动新增两个区块：

**1. 复核项统计（批次头）**
```
复核项统计: 共3项, 待处理2项, 已关闭1项
```

**2. 复核摘要表（按盒汇总）**

| 盒号 | 复核项总数 | 待处理数 | 已关闭数 | 复核项详情 |
|------|-----------|---------|---------|-----------|
| BOX-001 | 2 | 1 | 1 | #1[已关闭] 材料缺页:档A001缺3页; #3[处理中] 标签不清:档A002标签模糊 |
| BOX-002 | 1 | 1 | 0 | #2[待处理] 签收后补件:需补充合同附件 |

**3. 复核项明细表（如果存在复核项）**

包含：复核ID、盒号、问题类型、问题描述、责任方、截止时间、处理说明、状态、提报人、提报时间、关闭人、关闭时间

## 失败链路4：催办与升级处理

复核项创建后，接收方可对**待处理、处理中、已退回**状态的复核项发起催办，要求发送方尽快处理；发送方在催办待办列表中按紧急程度查看并回填处理结果。

### 接口契约与字段说明

**创建催办接口**：`POST /api/reviews/<review_id>/reminders`

请求参数（与其他接口统一使用 `operator_id`）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| operator_id | int | 是 | 操作员ID（必须是接收方） |
| reason | string | 是 | 催办原因 |
| expected_completion | string | 否 | 期望完成时间（日期格式） |
| urgency | string | 否 | 紧急程度：NORMAL/URGENT/CRITICAL，默认NORMAL |
| is_escalated | bool | 否 | 是否升级，默认false |

返回字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 催办ID |
| review_id | int | 关联复核项ID |
| reason | string | 催办原因（合并后可能包含多条） |
| urgency | string | 紧急程度 |
| is_escalated | int | 是否升级（0/1） |
| status | string | 状态：PENDING/PROCESSED/CANCELLED |
| merged | bool | 是否为合并结果（新增时为false，合并时为true） |
| created_by | int | 发起人ID |
| created_at | string | 创建时间 |
| processed_by | int | 处理人ID（已处理时有值） |
| processed_at | string | 处理时间（已处理时有值） |
| process_note | string | 处理备注（已处理时有值） |

### 催办字段

| 字段 | 说明 |
|------|------|
| 催办原因 | 必填，说明催办理由 |
| 期望完成时间 | 可选，期望何时处理完毕 |
| 紧急程度 | 普通(NORMAL) / 紧急(URGENT) / 特急(CRITICAL)，默认普通 |
| 是否升级 | 勾选后标记为升级催办，引起更高重视 |
| 处理备注 | 发送方回填的处理结果 |

### 催办状态

| 状态 | 说明 |
|------|------|
| 待处理(PENDING) | 催办已发起，等待发送方处理 |
| 已处理(PROCESSED) | 发送方已回填处理备注 |
| 已合并(MERGED) | 重复催办被合并到已有记录 |
| 已取消(CANCELLED) | 复核项关闭时催办自动收口 |

### 权限与操作

| 操作 | 权限角色 | 说明 |
|------|---------|------|
| 发起催办 | 接收方 | 对待处理/处理中/已退回的复核项发起 |
| 处理催办 | 发送方 | 回填处理备注，状态变为已处理 |
| 撤销升级 | 接收方 | 撤销最近一次升级，紧急程度降一级 |

### 统计口径

催办统计接口 `GET /api/reminders/stats` 返回：

| 字段 | 类型 | 说明 |
|------|------|------|
| total_pending | int | 待处理催办总数 |
| by_urgency | object | 按紧急程度分组：{NORMAL, URGENT, CRITICAL} |
| processed_today | int | 今日已处理数 |
| overdue_pending | int | 超期待处理数 |
| escalated_pending | int | 已升级待处理数 |

**统计规则**：
- 仅统计 `merged_into IS NULL` 的活跃催办（被合并的不重复计数）
- 待处理 = 状态为 PENDING
- 超期 = 关联复核项有截止时间且已超过截止日期且未关闭
- 已升级 = `is_escalated = 1` 且状态为 PENDING

### 重复催办冲突处理

同一条复核项上的催办采用**两级冲突处理**策略：

**第一级：短时间窗口拦截（60秒）**
- 同一用户在 60 秒内对同一复核项重复发起催办 → 明确拦截
- 返回 HTTP 429 状态码，提示"操作太频繁"
- 同时返回 `existing_reminder_id` 供前端定位已有记录

**第二级：合并策略（超过 60 秒后）**
- 已有待处理催办时，新催办合并到已有记录
- 原因追加（用分号连接）
- 紧急程度取更高者（NORMAL < URGENT < CRITICAL）
- 是否升级取"是"优先
- 期望完成时间取更晚的
- 合并操作写入催办日志（reminder_logs）

> 防止连续点击刷出一堆重复催办记录，同时保留合理的催办升级通道。

### 超期配置

复核项设置截止时间后，系统自动进行超期判断：

- **超期标记**：未关闭的复核项，截止时间早于当前日期 → 标记为"已超期"
- **超期统计**：催办统计中包含超期待办数量
- **超期展示**：催办待办卡片和复核项卡片上显示红色"已超期"标签

超期标记出现在：
- 批次详情的复核项卡片
- 催办待办列表的每条催办
- 催办统计汇总
- 导出的复核项明细

### 复核项生命周期与催办联动

| 复核项操作 | 催办联动 |
|-----------|---------|
| 关闭复核项 | 所有待处理催办自动取消(CANCELLED) |
| 撤销重开 | 已取消催办恢复为待处理(PENDING)，处理备注清空 |
| 退回复核项 | 已处理催办恢复为待处理(PENDING)，处理备注清空 |

### 催办待办列表

### 操作步骤

#### 步骤1：接收方发起催办

1. 切换当前用户为 `receiver`
2. 在复核项卡片中，对状态为待处理/处理中/已退回的复核项点击「🔔 催办」
3. 填写催办原因（必填）、期望完成时间（可选）、紧急程度（默认普通）、是否升级（可选）
4. 点击「发起催办」

> ⚠️ 校验规则：
> - 只有接收方才能发起催办
> - 已关闭和申请关闭状态的复核项不能催办
> - 60秒内重复提交会被拦截（429错误）
> - 超过60秒且已有待处理催办会自动合并

#### 步骤2：发送方处理催办

1. 切换当前用户为 `sender`
2. 点击顶部「催办待办」标签页查看所有待处理催办
3. 按紧急程度分组查看（特急优先），可看到超期和升级标记
4. 点击「处理」按钮，填写处理备注
5. 提交后催办状态变为已处理，处理结果回填到复核项明细

#### 步骤3：撤销升级（可选）

1. 接收方对已升级的催办点击「撤销升级」
2. 确认后 is_escalated 恢复为0，紧急程度降一级
3. 操作写入流转历史和催办日志

> ⚠️ 校验规则：
> - 只有接收方才能撤销升级
> - 只有已升级(is_escalated=1)的催办才能撤销

## 失败链路5：催办策略中心

值班人（接收方角色）可以在桌面端创建多套催办策略，定义触发条件、升级顺序、冷却时间、超时阈值、通知对象和适用范围。所有策略先通过预演验证效果，确认无误后再启用。策略数据持久化到SQLite，程序重启后继续生效。

### 核心设计理念

1. **先预演后启用**：每条规则启用前，先用样例数据模拟运行，直观看到哪些单据会命中、会被升到什么级别、会不会触发重复提醒
2. **优先级驱动的冲突裁决**：同一单据命中多条规则时，按优先级选择最优策略，同时保留完整的命中明细供审计
3. **版本化管理**：每次修改生成新版本快照，支持一键回滚到任意历史版本
4. **角色权限隔离**：只有接收方(RECEIVER)角色能管理策略，发送方(SENDER)完全无权限
5. **导入导出**：支持策略的批量迁移，格式校验和重复检测保障数据质量

### 策略数据模型

#### 核心字段

| 字段 | 类型 | 说明 |
|------|------|------|
| **name** | string | 策略名称（全局唯一） |
| **description** | string | 策略描述 |
| **priority** | int | 优先级（数值越大优先级越高，冲突时优先使用） |
| **status** | string | 状态：DRAFT(草稿)/ACTIVE(已启用)/INACTIVE(已停用) |
| **version** | int | 版本号，每次更新自动递增 |
| **trigger_conditions** | JSON | 触发条件配置 |
| **escalation_order** | JSON | 升级顺序数组（如：['普通催办', '紧急催办', '特急升级']） |
| **cooldown_minutes** | int | 冷却时间（分钟），防止重复催办 |
| **timeout_hours** | int | 超时阈值（小时），超过后自动升级 |
| **notify_targets** | JSON | 通知对象（角色数组：['SENDER', 'RECEIVER']） |
| **scope_filter** | JSON | 适用范围筛选（批次ID、盒子ID等） |

#### 触发条件 (trigger_conditions)

| 字段 | 类型 | 说明 |
|------|------|------|
| **issue_types** | array | 匹配的问题类型：['材料缺页', '标签不清', '签收后补件', '顺序混乱', '其他问题'] |
| **statuses** | array | 匹配的复核项状态：['OPEN', 'IN_PROGRESS', 'REJECTED'] |
| **min_hours_open** | int | 最低开放时长（小时），创建后超过此时长才会匹配 |
| **is_overdue** | bool | 是否超期：true/false/null（null表示不限） |

#### 数据库表结构

系统新增4张策略管理专用表：

```sql
-- 策略主表
CREATE TABLE reminder_strategies (
```

### 操作指南

#### 策略列表页面

以接收方身份登录后，顶部导航栏会出现「⚙️ 催办策略中心」标签页，点击进入策略管理界面。

**列表卡片展示：**

- 优先级和版本号
- 创建人、创建时间、更新人、更新时间
- 触发条件、升级顺序、冷却时间、超时阈值等核心配置
- 操作按钮：详情、预演、编辑、启用、停用

**状态说明：**
- 📝 **草稿(DRAFT)**：新创建的策略，可编辑、可预演、可启用
- ✅ **已启用(ACTIVE)**：正在运行的策略，自动生效，不能直接编辑
- ⏸️ **已停用(INACTIVE)**：已停止的策略，可重新编辑和启用

#### 新建策略

点击「+ 新建策略」进入创建表单：

**1. 基本信息**
- **策略名称**（必填，全局唯一）
- **策略描述**（可选，说明用途和适用场景）
- **优先级**（0-100，默认5，数值越大优先级越高）

**2. 触发条件配置**
- **问题类型**：勾选需要匹配的问题类型（材料缺页/标签不清/签收后补件/顺序混乱/其他问题）
- **复核项状态**：勾选需要匹配的状态（待处理/处理中/已退回）
- **最低开放时长**（小时）：创建后超过此时长才会匹配（0表示不限制）
- **是否超期**：不限/是（已超期）/否（未超期）

**3. 升级顺序**
- 点击「+ 添加升级级别」增加升级层级
- 按顺序填写每一级的说明（如：普通催办 → 紧急催办 → 特急升级）
- 至少保留1个升级级别

**4. 基础配置**
- **冷却时间**（分钟，必填）：两次催办之间的最小间隔，防止重复提醒
- **超时阈值**（小时，必填）：超过此时长自动升级到下一级别
- **通知对象**（角色）：勾选需要通知的角色（发送方/接收方）

**5. 适用范围**
- **批次ID列表**：逗号分隔，如：1,3,5（留空表示不限）
- **盒子ID列表**：逗号分隔，如：2,4,6（留空表示不限）
- **高级筛选（JSON）**：更复杂的筛选条件，如 `{"batch_ids": [1,2], "box_ids": [3]}`

**6. 保存与预演**
- 点击「🔍 预演效果」可直接预览当前配置的命中结果
- 点击「创建策略」保存为草稿状态

#### 策略预演（最核心的功能）

**两种预演方式：**
1. **已有策略预演**：在策略列表或详情页点击「🔍 预演」按钮
2. **草稿预演**：在编辑表单中点击「🔍 预演效果」，无需保存即可预览

**预演结果弹窗包含：**

**顶部统计面板（4项核心指标）：**
- 📊 **命中单据**：满足触发条件的复核项总数
- ⚠️ **冲突数量**：同时命中多条策略的复核项数量
- 📣 **将触发催办**：不在冷却期内、会实际触发催办的数量
- ⬆️ **将自动升级**：达到超时阈值、会自动升级的数量

**明细列表（每条命中的复核项）：**
- 复核项ID、问题类型、批次号、盒号、状态
- 紧急程度标签（普通/紧急/特急）
- 升级标签、冷却标签
- **冲突检测**：如果同时命中多条策略，显示橙色警告条，列出所有命中的策略名称和优先级
- **裁决结果**：显示最终选择的策略和裁决理由（如："选择'高优先级策略'(优先级100)，高于其他2条策略"）
- **升级级别**：显示当前升级级别和对应的升级步骤说明
- **冷却状态**：如果在冷却期内，显示剩余冷却时间
- **待处理催办**：显示该复核项已有的待处理催办数量和上次催办时间

> 💡 预演功能完全模拟真实运行逻辑，包括冲突裁决、冷却检测、升级计算，让你在启用前就能准确评估效果。

#### 冲突裁决机制

当同一复核项同时命中多条已启用的策略时，系统按以下规则裁决：

1. **优先级优先**：选择优先级最高的策略
2. **ID兜底**：优先级相同时，选择ID较大（较新）的策略
3. **完整记录**：
   - 所有命中的策略都会记录在 `matched_strategies` 字段
   - 最终选择的策略记录在 `selected_strategy` 字段
   - 裁决理由记录在 `resolution` 字段，便于审计

**示例裁决说明：**
```
同时命中3条策略，按优先级从高到低排序：
1. 高优先级策略(优先级100) ✓ 选中
2. 默认策略(优先级10)
3. 低优先级策略(优先级5)
```

#### 版本管理与回滚

**版本递增规则：**
- 新创建的策略版本为 v1
- 每次更新（PUT /api/strategies/<id>）版本号自动 +1
- 更新前自动保存当前版本的完整快照到 `reminder_strategy_snapshots` 表

**回滚操作：**
1. 在策略详情页点击「↩️ 回滚到上一版本」（仅非启用状态且版本>1时显示）
2. 确认后，策略配置恢复到上一版本
3. 版本号递减1
4. 操作记录到日志，可审计

**示例版本演进：**
```
v1: 创建策略 - 冷却时间60分钟
v2: 更新策略 - 冷却时间改为120分钟
v3: 更新策略 - 冷却时间改为180分钟
↓ 执行回滚
v2: 恢复到冷却时间120分钟（版本号回到v2）
```

#### 启用、停用与持久化

**启用流程：**
1. 点击「▶️ 启用」按钮
2. 状态从 DRAFT/INACTIVE 变为 ACTIVE
3. 操作记录到日志
4. 策略开始生效，自动匹配符合条件的复核项

**停用流程：**
1. 点击「⏸️ 停用」按钮
2. 确认后状态从 ACTIVE 变为 INACTIVE
3. 操作记录到日志
4. 策略停止生效

**SQLite持久化：**
- 所有策略数据（状态、版本、配置）实时写入 SQLite 数据库
- 程序重启后，从数据库恢复所有策略的状态和配置
- 已启用的策略继续保持 ACTIVE 状态，无需重新配置
- 直接查询数据库验证：
  ```sql
  SELECT id, name, status, version, cooldown_minutes, timeout_hours 
  FROM reminder_strategies;
  ```

#### 导入策略

1. 点击「📥 导入策略」标签页
2. 粘贴从其他系统导出的JSON数据
3. 系统自动进行三项校验：
   - **格式校验**：检查必填字段（name, trigger_conditions, escalation_order, cooldown_minutes, timeout_hours）
   - **重复检测**：检查导入文件内部是否有重复的策略名称
   - **冲突检测**：检查是否与现有策略名称重复
4. 校验通过后点击「开始导入」
5. 导入的策略默认为草稿状态，版本重置为 v1

**导入格式：**
```json
{
  "strategies": [
    {
      "name": "策略名称",
      "description": "策略描述",
      "priority": 10,
      "trigger_conditions": {
        "issue_types": ["材料缺页"],
        "statuses": ["OPEN"],
        "min_hours_open": 0,
        "is_overdue": null
      },
      "escalation_order": ["普通催办", "紧急催办"],
      "cooldown_minutes": 60,
      "timeout_hours": 24,
      "notify_targets": ["SENDER"],
      "scope_filter": {}
    }
  ]
}
```
