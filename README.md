# 档案移交流转台

本地档案移交流转系统，支持批次登记、装盒、移交、签收、退回、导出全流程可追溯。

## 快速启动

### 1. 安装依赖

```bash
pip install -r requirements.txt
```

### 2. 启动服务

```bash
python app.py
```

### 3. 访问系统

打开浏览器访问：http://127.0.0.1:5000/

## 默认账号

系统初始化时自动创建两个测试账号：

- `sender` - 发送方（负责登记、装盒、发起移交）
- `receiver` - 接收方（负责签收、退回、撤销退回）

在页面顶部的"当前用户"下拉菜单中切换身份。

## 操作流程

### 主流程（正常移交）

#### 步骤1：登记批次（发送方操作）

1. 点击"登记新批次"标签页
2. 输入**批次号**（全局唯一，如：DA-2025-001）
3. 输入批次描述（可选）
4. 添加档案：逐条输入**档号**（同一批次内不能重复）、**题名**、备注
5. 点击"登记批次"按钮

> ⚠️ 校验规则：
> - 批次号重复会被拒绝
> - 同一批次内档号重复会被拒绝，登记失败

#### 步骤2：装盒（发送方操作）

1. 在批次列表中点击刚创建的批次，进入详情页
2. 点击"+ 创建档案盒"，输入**盒号**（全局唯一，如：BOX-001）
3. 在档案盒卡片中勾选要装入的档案
4. 点击"确认装盒"
5. 可创建多个档案盒，分别装入不同档案
6. 当所有档案都装入盒子后，批次状态自动变为"已装盒"

> ⚠️ 校验规则：
> - 已装入其他盒子的档案不能再勾选
> - 移交后不能再修改装盒内容

#### 步骤3：发起移交（发送方操作）

1. 确认所有档案都已装盒
2. 点击"→ 发起移交"按钮
3. 确认提示后，批次和所有盒子状态变为"已移交"

#### 步骤4：签收（接收方操作）

1. **切换当前用户为 `receiver`（接收方）**
2. 进入批次详情页
3. 对每个档案盒点击"✓ 签收"按钮
4. 所有盒子签收完成后，批次状态变为"已签收"

> ⚠️ 校验规则：
> - 发送方不能代替接收方签收，会被拒绝
> - 必须使用接收方账号才能签收

#### 步骤5：导出移交清单

1. 点击"📥 导出移交清单"按钮
2. 自动下载 CSV 格式的移交清单文件
3. 导出记录会保存在"导出记录"区域，点击"重新下载"可随时下载对应时刻的历史快照，每份记录按导出记录独立编号存储，不会和其他批次或其他导出记录串号
4. 主批次"📥 导出移交清单"按钮下载的是该批次当前最新状态

导出清单包含：
- 批次基本信息（批次号、描述、创建人、状态、创建时间）
- 档案详情（档号、题名、备注、盒号、盒子状态）
- 完整流转历史（操作、操作人、时间、盒号、原因）

### 失败链路1：退回与撤销退回

#### 退回（接收方操作）

1. **可退回的状态**：盒子处于"已移交"或"已签收"状态时，接收方都可以选择退回
2. 点击"✗ 退回"按钮（已签收的盒子显示为"✗ 退回（签收后也可退）"）
3. **必须填写退回原因**（不能为空）
4. 确认后，盒子状态变为"已退回"，批次状态也变为"已退回"
5. **退回前快照自动保存**：系统会自动记录退回前的真实状态（已移交/已签收）、签收人、签收时间，写入流转历史，撤销退回时可精确还原

#### 撤销退回（接收方操作）

1. 盒子处于"已退回"状态时，可以撤销退回
2. 点击"↩ 撤销退回，恢复原状态"
3. **根据退回前快照精确恢复**：
   - 如果退回前是"已签收"：盒子恢复为"已签收"状态，签收人和原始签收时间都会还原，批次状态也会相应恢复
   - 如果退回前是"已移交"：盒子恢复为"已移交"状态，可重新签收
4. 批次状态智能聚合：
   - 如果所有盒子都撤销后都是"已签收" → 批次变为"已签收"
   - 如果有盒子是"已移交"或混合状态 → 批次变为"已移交"
   - 如果还有其他盒子是"已退回" → 批次保持"已退回"

> ⚠️ 校验规则：
> - 只有接收方才能退回和撤销退回
> - 退回原因不能为空
> - 撤销退回不会丢失签收记录，签收人、签收时间都从快照恢复

---

### 失败链路2：重复档号校验

在以下场景会校验重复档号：
1. 登记批次时，同一批次内有相同档号 → 登记被拒绝
2. 向已登记批次添加档案时，档号已存在 → 添加被拒绝

## 数据持久化

系统使用 SQLite 数据库（`archive_transfer.db` 文件）存储所有数据：

- ✅ 批次信息
- ✅ 档案信息
- ✅ 档案盒信息
- ✅ 档案与盒号映射关系
- ✅ 完整流转历史（经手人、时间戳、盒号、原因）
- ✅ 签收记录
- ✅ 退回原因
- ✅ 导出清单内容

**断电或重启后，所有数据完整保留**，重新启动服务即可继续操作，数据不会丢失。

---

## 流转历史记录

每个操作都会记录完整历史，包括：
- 操作类型（登记、装盒、移交、签收、退回、撤销退回、导出）
- 经手人及角色
- 精确时间戳
- 涉及的盒号
- 操作原因/备注

在批次详情页底部的时间线中可查看完整历史。

## 状态流转说明

### 批次状态

| 状态 | 说明 |
|------|------|
| 已登记 | 批次已创建，可添加档案和装盒 |
| 已装盒 | 所有档案已装入盒子，可发起移交 |
| 已移交 | 已发起移交，等待接收方处理 |
| 已签收 | 所有盒子已签收，流程完成 |
| 已退回 | 有盒子被退回（退回前可能是已移交或已签收） |

### 档案盒状态

| 状态 | 说明 | 可执行操作 |
|------|------|----------|
| 空盒 | 已创建，未装入档案 | 勾选档案、装盒 |
| 已装盒 | 已装入档案 | 等待移交 |
| 已移交 | 已发起移交 | 签收或退回（接收方操作） |
| 已签收 | 已被接收方签收 | 仍可退回（签收后也能退回，接收方操作） |
| 已退回 | 已被接收方退回 | 撤销退回，恢复退回前的真实状态 |

> **状态流转补充说明**：
> - 已移交 → 已签收（签收）
> - 已移交 → 已退回（退回）
> - 已签收 → 已退回（退回，签收后也可退）
> - 已退回 → 已移交（撤销退回，退回前是已移交的情况）
> - 已退回 → 已签收（撤销退回，退回前是已签收的情况，签收人、签收时间精确还原）

## 失败链路3：签收后异常复核（复核单管理）

批次到了"已签收"状态后，如果遇到**盒内材料缺页、标签不清、签收后补件**等异常情况，不再只能靠退回备注硬扛，可以通过「复核单」按盒子登记问题，全流程跟踪处理进度。

### 复核单字段

| 字段 | 说明 |
|------|------|
| 档案盒 | 问题发生在哪个盒子（必须从该批次的已有盒子中选择） |
| 问题类型 | 预设选项：材料缺页 / 标签不清 / 签收后补件 / 顺序混乱 / 其他问题 |
| 问题描述 | 详细描述问题，包括涉及的档号、具体情况 |
| 责任方 | 可选，指定负责处理该问题的部门/人员 |
| 截止时间 | 可选，要求处理完成的截止日期 |
| 处理说明 | 发送方填写的处理进展和处理结果 |
| 状态 | 待处理 → 处理中 → 申请关闭 → 已关闭；或 任何状态 → 已退回（需重处理） |

### 复核单状态流转与权限

**角色分工严格不交叉：**

| 操作 | 权限角色 | 说明 |
|------|---------|------|
| 新建复核项 | 接收方 | 按盒子登记问题，初始状态为「待处理」 |
| 更新处理结果 | 发送方 | 可改为「处理中」或「申请关闭」；申请关闭前必须填写处理说明 |
| 退回复核项 | 接收方 | 不认可处理结果时退回，状态变为「已退回」，退回原因必填 |
| 确认关闭 | 接收方 | 只有「申请关闭」状态才能确认关闭，状态变为「已关闭」 |
| 撤销重开 | 接收方 | 「已关闭」的复核项可撤销重开，原因必填；状态恢复为「待处理」，closed_by/closed_at 清空 |

**状态流转图：**

```
待处理(OPEN) ──发送方更新──► 处理中(IN_PROGRESS) ──发送方申请关闭──► 申请关闭(PENDING_CLOSE)
     │                              │                              │
     │                              │                              │接收方确认关闭
     │                              │                              ▼
     │退回                          │退回                       已关闭(CLOSED)
     │                              │                              │
     ▼                              ▼                              │接收方撤销重开
  已退回(REJECTED) ◄───────────────┘                               │
     │                                                              │
     └────────────── 发送方可重新更新为 处理中/申请关闭 ◄───────────┘
```

### 操作步骤

#### 步骤1：接收方新建复核项（接收方操作）

1. **切换当前用户为 `receiver`（接收方）**
2. 进入批次详情页，找到底部「📋 复核单管理」卡片
3. 点击右上角「+ 新建复核项」
4. 选择**档案盒**（该批次的所有已签收/已移交盒子都可选）
5. 选择**问题类型**，填写**问题描述**（必填）
6. 可选填写**责任方**和**处理截止时间**
7. 点击「提交复核项」

> ⚠️ 校验规则：
> - 只有接收方才能新建复核项，发送方会被拒绝
> - 问题类型 + 问题描述在同一盒子下重复且未关闭时，会返回 **409 冲突**，提示已有相同问题
> - 同一盒子下的不同问题、不同盒子的相同问题都允许正常提交
> - 提交后自动写入时间线（流转历史）

#### 步骤2：发送方更新处理结果（发送方操作）

1. **切换当前用户为 `sender`（发送方）**
2. 在复核单卡片中找到对应的复核项
3. 点击「✏️ 更新处理结果」
4. 选择更新后的状态：
   - **处理中**：正在处理，不需要接收方回应
   - **申请关闭**：已处理完成，等待接收方确认
5. 填写**处理说明**（申请关闭为必填）
6. 点击「提交更新」

> ⚠️ 校验规则：
> - 只有发送方才能更新处理结果，接收方会被拒绝
> - 已关闭的复核项不能直接更新，需要先撤销重开
> - 申请关闭必须填写处理说明，否则会被拒绝

#### 步骤3：接收方确认关闭 / 退回（接收方操作）

**确认关闭：**
1. 复核项状态为「申请关闭待确认」时，点击「✓ 确认关闭」
2. 确认后状态变为「已关闭」，记录关闭人和关闭时间

**退回（不认可处理）：**
1. 点击「✗ 退回（不认可处理）」
2. **必填退回原因**：说明不认可的地方或需要补充的内容
3. 状态变为「已退回（需重新提报）」，发送方需要重新更新处理结果

#### 步骤4：撤销重开已关闭项（接收方操作）

1. 找到已关闭的复核项，点击「↩ 撤销重开」
2. **必填撤销原因**：说明为什么需要重新处理
3. 状态恢复为「待处理」，关闭人和关闭时间清空
4. 发送方可以重新开始处理流程

> 💡 撤销重开后，完整的历史记录仍然保留在时间线中，可以追溯完整过程。

### 盒子卡片上的复核摘要

每个档案盒卡片的标题下方会显示复核摘要徽章：
```
📋 复核: 2待处理/1已关闭
```
- 待处理数 = 所有 OPEN + IN_PROGRESS + PENDING_CLOSE + REJECTED 状态的复核项
- 已关闭数 = CLOSED 状态的复核项
- 点击可跳转到复核单管理区查看详情

### 导出清单包含复核信息

导出移交清单时，自动新增两个区块：

**1. 复核项统计（批次头）**
```
复核项统计: 共3项, 待处理2项, 已关闭1项
```

**2. 复核摘要表（按盒汇总）**

| 盒号 | 复核项总数 | 待处理数 | 已关闭数 | 复核项详情 |
|------|-----------|---------|---------|-----------|
| BOX-001 | 2 | 1 | 1 | #1[已关闭] 材料缺页:档A001缺3页; #3[处理中] 标签不清:档A002标签模糊 |
| BOX-002 | 1 | 1 | 0 | #2[待处理] 签收后补件:需补充合同附件 |

**3. 复核项明细表（如果存在复核项）**

包含：复核ID、盒号、问题类型、问题描述、责任方、截止时间、处理说明、状态、提报人、提报时间、关闭人、关闭时间
